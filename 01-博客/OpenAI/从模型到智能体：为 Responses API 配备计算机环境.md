## 摘要

**1) 一句话总结**
OpenAI 通过 Responses API 为模型配备了包含 Shell 工具、托管容器、上下文压缩和可复用技能的计算机环境，使其能够安全、高效地执行复杂的现实世界智能体工作流。

**2) 关键要点**
*   **Responses API 编排**：API 在隔离的容器环境中自动编排模型与工具的交互循环，支持并发执行多个 Shell 命令并多路复用输出流。
*   **Shell 工具集成**：提供基于 Unix 的命令行工具（如 `grep`、`curl`），并支持运行 Go、Java 或 NodeJS 程序（需经过专门训练的 GPT-5.2 及更高版本模型）。
*   **输出限制机制**：为控制上下文消耗，API 会对 Shell 命令强制执行输出上限（例如限制为 1,000 个字符），截断中间部分并仅保留开头和结尾。
*   **原生上下文压缩**：提供服务器端内置或独立的 `/compact` 端点，将历史对话状态压缩为 Token 高效的加密表示形式，以支持长时间运行的任务（Codex 已应用此机制）。
*   **容器文件与数据库**：提供文件系统映射以暂存资源，并推荐使用 SQLite 等数据库进行结构化数据查询，从而替代将全量数据直接输入提示词的做法。
*   **受控的网络访问**：通过 Sidecar 出口代理管理出站网络请求，强制执行白名单和访问控制，并使用域范围的机密注入（模型仅可见凭据占位符）。
*   **智能体技能（Agent Skills）**：支持将重复的工作流模式打包为包含 `SKILL.md` 的版本化文件夹，API 会在执行前将其确定性地加载并解压到容器和模型上下文中。

**3) 风险与缺口**
*   **上下文溢出风险**：长时间运行的任务和庞大的 Shell 输出（如文件操作日志）会迅速耗尽上下文窗口预算，导致关键信息丢失。
*   **提示词过载**：将大型文件或整个电子表格直接打包到提示词中会增加成本，并使模型难以导航和提取信息。
*   **网络安全隐患**：赋予容器不受限制的互联网访问权限可能会将信息暴露给外部网站、无意中触及敏感的内部/第三方系统，并增加凭据泄露和数据外泄的风险。

## 正文

从模型到智能体：为 Responses API 配备计算机环境
============================================================================

作者：Bo Xu, Danny Zhang 和 Rohit Arunachalam

我们目前正处于一个转变期：从使用擅长特定任务的模型，转向使用能够处理复杂工作流的智能体。仅仅通过向模型提供提示词，你只能访问其训练出的智能。然而，为模型提供一个计算机环境，可以实现更广泛的用例，例如运行服务、从 API 请求数据，或生成更有用的产出物（如电子表格或报告）。

当你尝试构建智能体时，会出现一些实际问题：中间文件放在哪里？如何避免将大型表格直接粘贴到提示词中？如何在不引发安全隐患的情况下为工作流提供网络访问权限？以及如何在不自行构建工作流系统的情况下处理超时和重试？

我们没有让开发者自行构建执行环境，而是构建了必要的组件，为 [Responses API⁠(opens in a new window)](https://developers.openai.com/api/docs/guides/migrate-to-responses) 配备了计算机环境，以可靠地执行现实世界中的任务。

OpenAI 的 Responses API 结合 Shell 工具和托管的容器工作区，正是为了解决这些实际问题而设计的。模型提出步骤和命令；平台在一个隔离的环境中运行它们，该环境配备了用于输入和输出的文件系统、可选的结构化存储（如 SQLite）以及受限的网络访问权限。

在本文中，我们将详细介绍我们是如何为智能体构建计算机环境的，并分享一些关于如何利用它来实现更快、更可重复且更安全的生产工作流的早期经验。

Shell 工具
--------------

一个优秀的智能体工作流始于一个紧密的执行循环：模型提出一个动作（如读取文件或通过 API 获取数据），平台运行该动作，然后将结果反馈到下一步。我们将从 Shell 工具（观察此循环运行的最简单方式）开始，随后介绍容器工作区、网络、可复用技能以及上下文压缩。

要理解 Shell 工具，首先需要了解语言模型通常是如何使用工具的：例如调用函数或与计算机交互。在训练期间，模型会逐步看到工具的使用示例及其产生的效果。这有助于模型学习决定何时使用工具以及如何使用。当我们说“使用工具”时，我们的意思是模型实际上只是提出了一个工具调用请求。它无法自行执行该调用。

![图 1：Shell 工具“只是另一个工具”的图解](https://images.ctfassets.net/kftzwdyauwt9/1j8ghdf39iWtEXQAL8Wjf5/37b3a6136cb493574a71ffe483a03467/OAI_Equip_Responses_API_with_a_computer_environment_The_shell_tool_is_just_another_tool_desktop-light__3_.svg?w=3840&q=90)

Shell 工具使模型变得异常强大：它通过命令行与计算机交互，以执行各种任务，从搜索文本到在你的计算机上发送 API 请求。我们的 Shell 工具基于熟悉的 Unix 工具构建，开箱即用地提供了 `grep`、`curl` 和 `awk` 等实用程序，能够完成你所期望的任何操作。

与我们现有的仅执行 Python 的代码解释器相比，Shell 工具支持更广泛的用例，例如运行 Go 或 Java 程序，或启动 NodeJS 服务器。这种灵活性使模型能够完成复杂的智能体任务。

编排智能体循环
----------------------------

模型本身只能提出 Shell 命令，但这些命令是如何执行的呢？我们需要一个编排器来获取模型输出、调用工具，并将工具的响应在一个循环中传回给模型，直到任务完成。

Responses API 是开发者与 OpenAI 模型交互的方式。当与自定义工具一起使用时，Responses API 会将控制权交还给客户端，客户端需要自己的框架来运行这些工具。然而，该 API 也可以开箱即用地在模型和托管工具之间进行编排。

当 Responses API 接收到提示词时，它会组装模型上下文：用户提示词、先前的对话状态以及工具指令。为了使 Shell 执行生效，提示词必须提及使用 Shell 工具，**并且**所选模型必须经过训练以提出 Shell 命令——GPT-5.2 及更高版本的模型已为此进行了训练。有了所有这些上下文，模型随后决定下一步动作。如果它选择执行 Shell，它会向 Responses API 服务返回一个或多个 Shell 命令。API 服务将这些命令转发给容器运行时，以流式传输的方式返回 Shell 输出，并将其馈送到下一个请求的上下文中的模型。然后，模型可以检查结果、发出后续命令或生成最终答案。Responses API 会重复此循环，直到模型返回一个不再包含额外 Shell 命令的完成状态。

![图 2：智能体循环图解：Responses API 在容器中编排模型和 Shell 执行](https://images.ctfassets.net/kftzwdyauwt9/5XFQ2SSueNXO1UHX1Gx6J4/1065aa14107b9bb4682d44f0991cc832/OAI_Equip_Responses_API_with_a_computer_environment_Agent_loop__Responses_API_orchestrates_model_and_shell_execution_in_cont.svg?w=3840&q=90)

当 Responses API 执行 Shell 命令时，它会保持与容器服务的流式连接。随着输出的生成，API 会近乎实时地将其转发给模型，以便模型决定是等待更多输出、运行另一个命令，还是继续生成最终响应。

![图 3：流式传输 Shell 命令执行输出](https://images.ctfassets.net/kftzwdyauwt9/5TOlwyV32C5GOvMzLCcuSq/2210a01520870c4e688fe27aece58d85/OAI_Equip_Responses_API_with_a_computer_environment_Streaming_shell_command_execution_output_desktop-light__3_.svg?w=3840&q=90)

*Responses API 流式传输 Shell 命令输出*

模型可以在一个步骤中提出多个 Shell 命令，Responses API 可以使用独立的容器会话并发执行它们。每个会话独立地流式传输输出，API 将这些流多路复用回结构化的工具输出中作为上下文。换句话说，智能体循环可以并行处理工作，例如搜索文件、获取数据和验证中间结果。

![图 4：Responses API 多路复用命令执行会话](https://images.ctfassets.net/kftzwdyauwt9/2gn7fUz2H2FbSbBsMYa58G/87622df28230c97e9d4cd29aec54380c/OAI_Equip_Responses_API_with_a_computer_environment_Responses_API_multiplexes_command_execution_sessions_desktop-light__3_.svg?w=3840&q=90)

当命令涉及文件操作或数据处理时，Shell 输出可能会变得非常大，并在不增加有用信号的情况下消耗上下文预算。为了控制这一点，模型会为每个命令指定一个输出上限。Responses API 会强制执行该上限，并返回一个有界的结果，该结果保留了输出的开头和结尾，同时标记了省略的内容。例如，你可能会将输出限制为 1,000 个字符，并保留开头和结尾：

`开头的文本 ... 截断了 1000 个字符 ... 结尾的文本`

并发执行和有界输出共同使智能体循环既快速又具有上下文效率，因此模型可以继续对相关结果进行推理，而不会被原始终端日志所淹没。

当上下文窗口变满时：压缩
---------------------------------------------

智能体循环的一个潜在问题是任务可能会运行很长时间。长时间运行的任务会填满上下文窗口，而上下文窗口对于在多轮对话和多个智能体之间提供上下文至关重要。想象一下，一个智能体调用一项技能、获得响应、添加工具调用和推理摘要——有限的上下文窗口很快就会被填满。为了避免在智能体继续运行时丢失重要的上下文，我们需要一种方法来保留关键细节并移除任何无关内容。我们没有要求开发者设计和维护自定义的摘要或状态传递系统，而是在 Responses API 中添加了原生压缩功能，旨在与模型的行为和训练方式保持一致。

我们最新的模型经过训练，能够分析先前的对话状态，并生成一个压缩项，该压缩项以加密且 Token 高效的表示形式保留了关键的先前状态。压缩后，下一个上下文窗口由该压缩项和早期窗口中的高价值部分组成。这使得工作流能够跨越窗口边界连贯地继续进行，即使在扩展的多步骤和工具驱动的会话中也是如此。[Codex 依赖于这种机制](https://openai.com/index/unrolling-the-codex-agent-loop/)来维持长时间运行的编码任务和迭代的工具执行，而不会降低质量。

压缩功能既可以作为服务器上的内置功能使用，也可以通过独立的 `/compact` 端点使用。服务器端压缩允许你配置一个阈值，系统会自动处理压缩时机，从而消除了复杂的客户端逻辑需求。它允许一个稍大的有效输入上下文窗口，以容忍压缩前的小幅超量，因此接近限制的请求仍然可以被处理和压缩，而不是被拒绝。随着模型训练的发展，原生压缩解决方案也会随着每个 OpenAI 模型的发布而不断演进。

Codex 帮助我们构建了压缩系统，同时也是该系统的早期用户。当一个 Codex 实例遇到压缩错误时，我们会启动第二个实例进行调查。结果是，Codex 仅仅通过解决这个问题就获得了一个原生的、有效的压缩系统。Codex 这种自我检查和完善的能力，已经成为在 OpenAI 工作中一个特别有趣的部分。大多数工具只需要用户学习如何使用它们；而 Codex 则与我们一起学习。

容器上下文
-----------------

现在让我们来讨论状态和资源。容器不仅是运行命令的地方，也是模型的工作上下文。在容器内部，模型可以读取文件、查询数据库，并在网络策略控制下访问外部系统。

![图 5：展示运行时容器内部的图解：文件、数据库、技能和受策略控制的网络](https://images.ctfassets.net/kftzwdyauwt9/2rdxS5bXiQLnB1Vg05psLC/2ed0d4804552378d2b84afa12ce61b28/OAI_Equip_Responses_API_with_a_computer_environment_Inside_the_runtime_container_desktop-light__3_.svg?w=3840&q=90)

#### 文件系统

容器上下文的第一部分是用于上传、组织和管理资源的文件系统。我们构建了 [容器和文件⁠(opens in a new window)](https://developers.openai.com/api/reference/resources/containers) API，为模型提供可用数据的映射，并帮助其选择有针对性的文件操作，而不是执行广泛且嘈杂的扫描。

一个常见的反模式是将所有输入直接打包到提示词上下文中。随着输入的增加，过度填充提示词会变得昂贵，并且使模型难以导航。更好的模式是将资源暂存在容器文件系统中，让模型决定使用 Shell 命令打开、解析或转换哪些内容。就像人类一样，模型在处理有组织的信息时表现更好。

#### 数据库

容器上下文的第二部分是数据库。在许多情况下，我们建议开发者将结构化数据存储在 SQLite 等数据库中并进行查询。例如，与其将整个电子表格复制到提示词中，不如向模型提供表格的描述（存在哪些列以及它们的含义），让它提取所需的行。

例如，如果你问：“本季度哪些产品的销量下降了？”模型可以**仅**查询相关的行，而不是扫描整个电子表格。这更快、更便宜，并且更容易扩展到更大的数据集。

#### 网络访问

容器上下文的第三部分是网络访问，这是智能体工作负载的重要组成部分。智能体工作流可能需要获取实时数据、调用外部 API 或安装包。与此同时，赋予容器不受限制的互联网访问权限可能会带来风险：它可能会将信息暴露给外部网站，无意中触及敏感的内部或第三方系统，或者使凭据泄露和数据外泄更难防范。

为了在不限制智能体实用性的情况下解决这些问题，我们构建了使用 Sidecar 出口代理的托管容器。所有出站网络请求都流经一个集中的策略层，该策略层强制执行白名单和访问控制，同时保持流量的可观测性。对于凭据，我们在出口处使用域范围的机密注入。模型和容器只能看到占位符，而原始机密值保留在模型可见的上下文之外，并且仅应用于批准的目的地。这降低了泄露的风险，同时仍然允许经过身份验证的外部调用。

智能体技能
------------

Shell 命令非常强大，但许多任务都会重复相同的多步模式。智能体在每次运行时都必须重新发现工作流——重新计划、重新发出命令并重新学习约定——这会导致结果不一致和执行浪费。[智能体技能（Agent skills）⁠(opens in a new window)](https://agentskills.io/home) 将这些模式打包成可复用、可组合的构建块。具体来说，一项技能是一个文件夹包，其中包含“[SKILL.md⁠(opens in a new window)](http://skill.md/)”（包含元数据和指令）以及任何支持资源，例如 API 规范和 UI 资产。

这种结构自然地映射到我们前面描述的运行时架构。容器提供持久化文件和执行上下文，而 Shell 工具提供执行接口。有了这两者，模型可以在需要时使用 Shell 命令（如 `ls`、`cat` 等）发现技能文件，解释指令，并在同一个智能体循环中运行技能脚本。

我们提供了 [API⁠(opens in a new window)](https://developers.openai.com/api/reference/resources/skills) 来管理 OpenAI 平台中的技能。开发者将技能文件夹作为版本化的包上传并存储，稍后可以通过技能 ID 检索。在将提示词发送给模型之前，Responses API 会加载该技能并将其包含在模型上下文中。这个序列是确定性的：

1.   获取技能元数据，包括名称和描述。
2.   获取技能包，将其复制到容器中并解压。
3.   使用技能元数据和容器路径更新模型上下文。

在决定某项技能是否相关时，模型会逐步探索其指令，并通过容器中的 Shell 命令执行其脚本。

智能体是如何构建的
-------------------

将所有部分组合在一起：*Responses API* 提供编排，*Shell 工具* 提供可执行的动作，*托管容器* 提供持久的运行时上下文，*技能* 叠加了可复用的工作流逻辑，而 *压缩* 则允许智能体带着所需的上下文长时间运行。

有了这些原语，一个单一的提示词就可以扩展为一个端到端的工作流：发现正确的技能、获取数据、将其转换为本地结构化状态、高效地查询它，并生成持久的产出物。
