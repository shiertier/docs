## 摘要

### 1) 一句话总结
本文探讨了如何通过组合使用 Claude Skills（懒加载专业知识）、MCP（工具能力）和子代理（上下文隔离），构建低成本、高效率的 AI 代理基础设施，从而彻底摆脱重复繁琐的提示词工程。

### 2) 核心要点
* **Claude Skills 的懒加载机制**：Skills 是存放在特定目录下的可复用指令集（`skill.md`），采用三层渐进式披露（元数据约 100 Tokens、主体约 5,000 Tokens、按需引用的文件），仅在相关时加载，避免了上下文臃肿。
* **MCP 的高昂开销**：传统的 MCP（模型上下文协议）采用“急切加载”模式，无论是否使用，都会将所有工具元数据（如典型的 32,000 个 Token）注入每条消息，导致高成本（如每月约 160 美元纯开销）、高延迟和输出质量下降。
* **能力与专业知识的互补**：MCP 赋予代理与外部系统交互的“能力”（做什么），而 Skills 赋予代理遵循特定规范和工作流的“专业知识”（怎么做），两者结合才能实现高效自动化。
* **子代理（Subagents）的上下文隔离**：子代理是拥有独立上下文和专属工具的专用工作节点。它们在完成任务后会丢弃中间推理过程和工具元数据，只将结果返回给主代理，从而保持主代理上下文的干净和廉价。
* **架构最佳实践**：主代理应作为“执行线程”负责编排，将具体任务委托给配备了特定 MCP 工具和 Skill 的子代理（例如将 GitHub MCP 和 PR 规范 Skill 组合，交由子代理完成代码合并请求）。
* **AI 基础设施的范式转移**：AI 开发的重心已从单纯提升模型智力，转向构建更好的基础设施（技能、子代理、多代理编排），以实现代理式 AI 规模化应用的经济和操作可行性。
* **代理即函数（Agents as Functions）**：代理正在演变为软件工程中的“新函数”，具备独立的输入、内部状态、工具、指令和输出，支持复用、组合、可观测性以及故障隔离。

### 3) 风险与不足
* **并发写入与共享状态问题**：多代理并行处理在“读取”任务中表现良好，但在涉及共享状态的“写入”任务时极易出错（例如前后端子代理并发修改导致代码不一致），这在当前仍是一个未解决的难题。
* **代理间通信协议采用缓慢**：虽然 Google 的 A2A（代理到代理协议）旨在标准化代理间的直接通信，但与 MCP 的爆发式增长相比，其采用率一直非常缓慢，目前尚不具备大规模押注的价值。

## 正文

* * *

_本文反映了截至 2026 年 2 月 Claude Skills、MCP 和子代理的发展状况。AI 发展迅速，因此当您阅读本文时，某些细节可能已经过时。然而，本文重点探讨的核心概念是永恒的。_

* * *

如果您使用 LLM 进行开发已经有一段时间了，您可能一遍又一遍地经历过这样的循环：您花时间精心编写了一个能产生出色结果的提示词（prompt），几天后您又需要同样的行为，于是您又从头开始编写提示词。重复几次后，您也许意识到了这种低效，于是打算把提示词模板存到某个地方以便日后调用。但即便如此，您还是需要找到那个提示词，把它粘贴进去，并针对当前的对话进行微调。这太繁琐了。

这就是我所说的_提示词工程仓鼠轮_。这是一个从根本上就存在缺陷的工作流。

Claude Skills 是 Anthropic 对这个“可复用提示词”问题的解答，_而且远不止于此_。除了让您免受重复编写提示词的折磨，它们还引入了一种截然不同的方法来处理上下文管理、Token 经济学以及 AI 驱动的开发工作流架构。

在本文中，我将详细解析技能（Skills）和子代理（Subagents）究竟是什么，它们与传统的 MCP 有何不同，以及 技能 / MCP / 子代理 的组合将走向何方。

* * *

什么是_技能（Skills）_？
------------------

从核心上讲，技能是可复用的指令集，像 Claude 这样的 AI 代理可以在它们与对话相关时自动访问它们。您只需编写一个包含一些元数据和指令主体的 `skill.md` 文件，将其放入 `.claude/skills/` 目录中，剩下的就交给 Claude 了。

### 它们的外观

在最简单的形式下，技能就是一个包含名称、描述和指令主体的 Markdown 文件，如下所示：

```markdown
---

name: <skill-name>

description: <short-skill-description>

---

<skill-details>
```

### 它们的优势

技能的主要优势在于自动调用。在开始新对话时，代理只会读取每个技能的名称和描述，以节省 Token。当它确定某个技能相关时，才会加载主体内容。如果主体引用了其他文件或文件夹，代理也会读取它们，但仅在它认为需要时才读取。本质上，技能是懒加载（lazy-loaded）的上下文。代理不会预先消耗完整的指令集。它会向自己渐进式披露（progressively discloses）信息，只提取当前步骤所需的内容。

这种渐进式披露在三个层级上运作，每个层级都有自己的上下文预算：

1.   **元数据（启动时加载）：** 技能的名称（最多 64 个字符）和描述（最多 1,024 个字符）。每个技能大约消耗 ~100 个 Token，即使注册了数百个技能，这种开销也微乎其微。
2.   **技能主体（调用时加载）：** `skill.md` 内部的完整指令集，最多约 5,000 个 Token。只有当代理确定该技能相关时，这部分才会进入上下文窗口。
3.   **引用的文件（按需加载）：** 技能目录中的其他 Markdown 文件、文件夹或脚本。这里几乎没有限制，代理会按需读取这些内容，只有当指令引用它们且当前任务需要时才会读取。

![Image 1](https://contributor.insightmediagroup.io/wp-content/uploads/2026/02/skill_levels_diagram-1024x597.webp)

技能在三个层级上渐进式加载上下文：技能摘要（元数据）、主体（详细指令）和引用的文件（附加上下文），每一层仅在需要时触发。

> **洞察：** 技能是可复用、懒加载且自动调用的指令集，它在元数据、主体和引用文件三个层级上使用渐进式披露。这避免了将所有内容一股脑塞进上下文窗口，从而将前期成本降至最低（说的就是你，MCP ![Image 2: 👀](https://s.w.org/images/core/emoji/17.0.2/svg/1f440.svg)）。

* * *

Token 经济学中的问题
------------------------------

### 成本因素

这不是什么秘密；代理的上下文窗口空间不是免费的，填满它会产生复合成本。上下文窗口中的每个 Token 都会在三个方面让您付出代价：

1.   **实际成本：** 最明显的一点是您需要按 Token 付费。这可以直接通过 API 使用量体现，也可以间接通过使用限制体现。
2.   **延迟：** 您还在用时间买单，因为输入 Token 越多，响应速度就越慢。这与上下文窗口的长度（~注意力机制）无法很好地成比例扩展。
3.   **质量：** 最后，长上下文窗口还会导致质量下降。事实证明，当 LLM 的上下文中充斥着无关信息时，其表现会变差。

### MCP 昂贵的开销

让我们通过一个简单的粗略计算来直观地了解这一点。我在编程时常用的 MCP 选择包括：

*   **AWS** 用于基础设施部署。三个服务器（`aws-mcp`、`aws-official`、`aws-docs`）加起来大约消耗 ~8,500 个 Token（13 个工具）。
*   **Context7** 用于文档。元数据大约 ~750 个 Token（2 个工具）。
*   **Figma** 用于将设计引入前端开发。元数据大约 ~500 个 Token（2 个工具）。
*   **GitHub** 用于在其他仓库中搜索代码。元数据大约 ~2,000 个 Token（26 个工具）。
*   **Linear** 用于项目管理。元数据大约 ~3,250 个 Token（33 个工具）。
*   **Serena** 用于代码搜索。元数据大约 ~4,500 个 Token（26 个工具）。
*   **Sentry** 用于错误追踪。元数据大约 ~12,500 个 Token（22 个工具）。

这总计大约是 **~32,000 个 Token** 的工具元数据，无论您是否与该工具交互，它们都会被加载到_每一条消息_中。

用具体的金额来衡量：Claude Opus 4.6 每百万输入 Token 收费 5 美元。这 32K 闲置的 MCP 元数据 Token 会为您发送的**每一条消息增加 0.16 美元**的成本。听起来很少，直到您意识到，即使是一个简单的 5 条消息的对话，也已经增加了 0.8 美元的纯开销。而大多数开发者发送的消息远不止 5 条；加上一些简短的澄清和收集上下文的问题，您很快就会达到几十甚至上百条消息。假设您在一个 20 天的工作月里平均每天发送 50 条消息，那就是每天 8 美元，**每月 ~$160*** 的_纯开销_，仅仅是为了让工具描述停留在上下文中。这还没算上对延迟和质量的影响。

> ***一个小注脚：** 大多数模型对缓存的输入 Token 收费要低得多（90% 的折扣）。这个注脚的注脚是，其中一些模型在启用缓存时会额外收费，而且它们并不总是默认启用（API）缓存（咳咳，_Claude_，咳咳）。

### **技能的高性价比方法**

技能的加载模式从根本上改变了所有这三个成本因素。一开始，代理只看到每个技能的名称和简短描述，每个技能大约 ~100 个 Token。这样一来，即使我注册了 300 个技能，消耗的 Token 仍然比我的 MCP 设置要少。完整的指令主体（~5,000 个 Token）只有在代理认为相关时才会加载，而引用的文件只有在当前步骤需要时才会加载。

在实践中，一次典型的对话可能只会调用一两个技能，而其余的对上下文窗口保持不可见。这就是关键的区别：MCP 的成本随着_已注册_工具（跨所有服务器）的数量而增加，而技能的成本则更紧密地与实际使用量挂钩。

![Image 3](https://contributor.insightmediagroup.io/wp-content/uploads/2026/02/mcp_eager_vs_skills_lazy-1024x529.webp)

MCP 预先加载所有元数据。技能仅在相关时加载上下文，这种差异在每条消息中都会产生复合影响。

> **洞察：** MCP 是“急切的（eager）”，无论是否使用，都会预先加载所有工具元数据。技能是“懒惰的（lazy）”，仅在相关时渐进式加载上下文。这种差异对成本、延迟和输出质量至关重要。

### **等等，这有误导性？技能和 MCP 完全是两码事！**

如果上面的内容读起来像是“技能是全新且更好的 MCP”，那么请允许我纠正这种说法。我的本意是聚焦于它们的加载模式以及它们对 Token 消耗的影响。在功能上，它们截然不同。

MCP（模型上下文协议）是一个开放标准，赋予任何 LLM 与外部应用程序交互的能力。在 MCP 之前，将 `M` 个模型连接到 `N` 个工具需要 `M * N` 个自定义集成。MCP 将其简化为 `M + N`：每个模型实现一次该协议，每个工具暴露一次该协议，然后它们就可以相互操作了。这是一个简单的基础设施变革，但它确实非常强大（难怪它风靡全球）。

另一方面，技能在某种程度上是“美化版的提示词”，我是往最好的方面说。它们赋予代理专业知识和方向，告诉它如何处理任务、遵循什么约定、何时使用哪个工具，以及如何构建其输出。它们是按需获取的可复用指令集，仅此而已。

> **洞察：** MCP 赋予代理能力（“做什么”）。技能赋予它专业知识（“怎么做”），因此它们是互补的。

举个例子让这一点更具体。假设您将 GitHub 的 MCP 服务器连接到您的代理。MCP 赋予代理创建拉取请求（PR）、列出 Issue 和搜索仓库的能力。但它并没有告诉代理，例如，您的团队如何构建 PR、您总是包含一个测试部分、您按变更类型进行标记、您在标题中引用 Linear 工单。这正是技能的作用。MCP 提供工具，技能提供剧本（playbook）。

因此，当我在前面展示技能比 MCP 更高效地加载上下文时，真正的结论并不是“用技能代替 MCP”，而是 **懒加载作为一种模式是行之有效的**。因此，值得一问：为什么 MCP 工具访问不能也进行懒加载呢？这就是子代理（subagents）发挥作用的地方。

* * *

**子代理：两全其美**
----------------------------------

子代理是专门的_子级代理_，拥有自己独立的上下文窗口和连接的工具。有两个特性使它们非常强大：

*   **隔离的上下文：** 子代理从一个干净的上下文窗口开始，预先加载了它自己的系统提示词，并且只有分配给它的工具。它读取、处理和生成的所有内容都保留在自己的上下文中，主代理（main agent）只看到最终结果。
*   **隔离的工具：** 每个子代理都可以配备自己的一套 MCP 服务器和技能。主代理不需要知道（也不需要为之付费）它从未直接使用的工具。

一旦子代理完成任务，它的整个上下文就会被丢弃。工具元数据、中间推理过程、API 响应：统统消失。只有结果流回主代理。这实际上是一件大好事。我们不仅避免了用不必要的工具元数据使主代理的上下文变得臃肿，还防止了不必要的推理 Token 污染上下文。举个说明性的例子，想象一个研究库 API 的子代理。它可能会在多个文档源中搜索，阅读几十页内容，并尝试多次查询，然后才找到正确的答案。您仍然需要为子代理自身的 Token 使用量付费，但所有这些中间工作、死胡同、不相关的页面、搜索查询，都会在子代理完成任务后被丢弃。关键的好处是，这些都不会复合叠加到主代理的上下文中，因此您对话中的每一条后续消息都保持干净且廉价。

这意味着您可以这样设计您的设置：MCP 服务器只能通过特定的子代理访问，而永远不会加载到主代理上。主代理不再需要在每条消息中携带 ~32,000 个 Token 的工具元数据，而是几乎携带零元数据。当它需要打开一个拉取请求时，它会启动一个 GitHub 子代理，创建 PR，并返回链接。就像技能是懒加载的_上下文_一样，子代理是懒加载的_工作者_：主代理知道它可以调用哪些专家，并且只有在任务需要时才会启动一个。

### 一个实际案例

让我们把它变得更具体。我每天使用的一个工作流是“特性分支收尾（feature branch wrap-up）”，它自动化了我开发周期中非常繁琐的一部分：打开拉取请求（PR）。以下是技能、MCP 和子代理如何协同工作的。

在主代理和我完成编码工作后，我要求它对特性分支进行收尾。主代理本身不处理这个问题；它将整个 PR 工作流委托给一个专用的子代理。这个子代理配备了 GitHub MCP 服务器和一个 `change-report` 技能，该技能定义了我的团队如何构建 PR。它的 `skill.md` 大致如下所示：

```markdown
---
name: change-report
description: 在为 PR 生成变更报告时使用。
   定义了团队的 PR 结构、分类规则和格式约定。
---

1. 确保没有遗留的暂存变更，否则向主代理报告。
2. 运行 `git diff dev...HEAD --stat` 和 `git log dev..HEAD --oneline`
   以收集此特性分支上的所有变更。
3. 分析差异，并按类型（新特性、重构、错误修复或配置变更）对最关键的变更进行分类。
4. 按照 `pr-template.md` 中的模板生成结构化的变更报告。
5. 通过 GitHub MCP 打开 PR，使用生成的报告填充标题和正文。
6. 回复 PR 链接。
```

同一目录下的 `pr-template.md` 文件定义了我团队的 PR 结构：摘要、变更明细和测试说明部分。这是渐进式披露的第 3 层：子代理只有在第 4 步告诉它时才会读取它。

这就是该设置能够奏效的原因。技能提供了关于我的团队如何报告变更的专业知识，GitHub MCP 提供了实际创建 PR 的能力，而子代理提供了执行所有这些工作的上下文边界。另一方面，主代理只负责调用子代理，等待它完成，然后收到确认信息或出错提示。

![Image 4](https://contributor.insightmediagroup.io/wp-content/uploads/2026/02/skills_mcp_subagents_architecture-1024x666.webp)

实际运行中的 PR 工作流：主代理将整个 PR 流程委托给一个配备了 change-report 技能和 GitHub MCP 访问权限的子代理。

> **洞察：** 技能、MCP 和子代理和谐地协同工作。技能提供专业知识和指令，MCP 提供能力，子代理提供上下文边界（保持主代理的上下文干净）。

**宏观视角**
----------------------

在 LLM 的早期阶段，竞争的焦点是更好的模型：更少的幻觉、更敏锐的推理、更具创造性的输出。这场竞赛并没有完全停止，但重心显然已经转移。MCP 和 Claude Code 是真正具有革命性的。老实说，将 Claude Sonnet 从 3.5 升级到 3.7 并非如此。我们今天获得的渐进式模型改进，其重要性远不及我们围绕它们构建的基础设施。技能、子代理和多代理编排都是这种转变的一部分：从“我们如何让模型变得更聪明”转变为“我们如何从现有的东西中获取最大价值”。

> **洞察：** AI 开发的价值已经从更好的模型转移到了更好的基础设施。技能、子代理和多代理编排不仅仅是开发者体验的改进；它们是使代理式 AI（agentic AI）在规模化应用中具有经济和操作可行性的架构。

### 我们今天所处的位置

技能通过将您最好的提示词转化为可复用、自动调用的指令集，解决了提示词工程仓鼠轮的问题。子代理通过将工具访问和中间推理隔离到专用的工作者中，解决了上下文臃肿的问题。它们结合在一起，使您能够一次性将专业知识代码化，并自动应用于未来的每一次交互中。这正是遵循行业最佳实践的工程团队在文档、样式指南和运行手册（runbooks）方面已经在做的事情。技能和子代理只是让这些工件变得机器可读。

子代理模式也正在解锁多代理并行性。您可以同时启动多个子代理，让它们独立工作并收集它们的结果，而不是让一个代理按顺序处理任务。Anthropic 自己的[多代理研究系统](https://www.anthropic.com/engineering/multi-agent-research-system)已经做到了这一点：Claude Opus 4.6 负责编排，而 Claude Sonnet 4.6 子代理并行执行。这自然引出了异构模型路由（heterogeneous model routing），即由昂贵的前沿模型负责编排和计划，而较小、较便宜的模型负责执行。编排者负责推理，工作者负责执行。这可以在保持输出质量的同时大幅降低成本。

这里有一个重要的注意事项。虽然并行性在_读取_任务中表现良好，但在涉及共享状态的_写入_任务中却困难得多。例如，假设您并行启动了一个后端和一个前端子代理。后端代理重构了一个 API 端点，而前端代理基于该变更前获取的快照，生成了调用旧端点的代码。孤立地看，这两个代理都没有错，但它们合在一起却产生了一个不一致的结果。这是一个经典的并发问题，源于不久的将来的 AI 工作流，至今仍是一个未解决的难题。

### **未来的发展方向**

我预计技能的组合将变得更加复杂。如今，技能相对扁平：一个带有可选引用的 Markdown 文件。但该架构自然支持引用其他技能的分层技能，从而创建类似专业知识继承层次结构的东西。想象一下，一个基础的“代码审查”技能被特定语言的变体扩展，然后再被特定团队的约定进一步扩展。

如今的大多数多代理系统都是严格分层的：主代理委托给子代理，子代理完成任务，然后控制权返回。目前子代理之间还没有太多的点对点协作。Anthropic 最近为 Opus 4.6 推出的[“代理团队（agent teams）”功能](https://code.claude.com/docs/en/agent-teams)是朝着这个方向迈出的早期一步，它允许多个代理直接协调，而不是通过编排者路由所有内容。在协议方面，Google 的 A2A（Agent-to-Agent Protocol，代理到代理协议）可以跨提供商标准化这种模式；MCP 处理代理到工具的通信，而 A2A 将处理代理到代理的通信。话虽如此，与 MCP 的爆炸性增长相比，A2A 的采用一直很缓慢。这是一个值得关注的领域，但目前还不值得押注。

### **代理将成为新的函数**

这里出现了一个更广泛的抽象概念，值得我们退一步来欣赏。Andrej Karpathy 著名的推文_“最热门的新编程语言是英语”_捕捉到了我们与 LLM 交互方式的真实本质。但技能和子代理将这种抽象又推进了一层：**代理正在成为新的函数（functions）**。

子代理是一个自包含的工作单元：它接收输入（任务描述），拥有自己的内部状态（上下文窗口），使用特定的工具（MCP 服务器），遵循特定的指令（技能），并返回输出。它可以从多个地方被调用，它是可复用的，也是可组合的。这就是一个函数。主代理变成了执行线程：负责编排、分支、委托，并综合来自专业工作者的结果。

除了这个比喻之外，它还能产生与函数对软件工程相同的实际影响。隔离限制了代理失败时的爆炸半径，而不是破坏整个系统，并且可以通过 try-except 机制捕获故障。专业化意味着每个代理都可以针对其特定任务进行优化。可组合性意味着您可以从简单的、可测试的部件构建越来越复杂的工作流。可观测性也自然而然地随之而来；由于每个代理都是具有明确输入和输出的离散单元，追踪_“为什么系统会执行操作 X”_变成了检查调用栈（call stack），而不是盯着 20 万 Token 的上下文转储发呆。

![Image 5](https://contributor.insightmediagroup.io/wp-content/uploads/2026/02/agents_as_functions-1024x563.webp)

子代理直接映射到函数：输入、内部状态、工具、指令和输出。主代理是执行线程。

**结论**
--------------

技能表面上看起来像是简单的“可复用提示词”，但它们实际上代表了对 AI 工具中一些最困难问题的深思熟虑的解答：上下文管理、Token 效率，以及原始能力与领域专业知识之间的差距。

如果您还没有尝试过技能，可以从小处着手。挑选您最常重复的提示词模式，将其提取到 `skill.md` 中，看看它如何改变您的工作流。一旦您掌握了这一点，就可以迈出下一步：确定哪些 MCP 工具不需要驻留在您的主代理上，或者哪些子流程需要大量推理（这些推理在找到答案后就不再需要了），并将它们的作用域限定在专用的子代理中。当每个代理只携带它真正需要的东西时，您的设置会变得多么干净，这会给您带来惊喜。

### **本文的核心洞察**

*   技能是可复用、懒加载且自动调用的指令集，它在元数据、主体和引用文件三个层级上使用渐进式披露。这避免了将所有内容一股脑塞进上下文窗口，从而将前期成本降至最低（说的就是你，MCP ![Image 6: 👀](https://s.w.org/images/core/emoji/17.0.2/svg/1f440.svg)）。
*   MCP 是“急切的”，无论是否使用，都会预先加载所有工具元数据。技能是“懒惰的”，仅在相关时渐进式加载上下文。这种差异对成本、延迟和输出质量至关重要。
*   MCP 赋予代理能力（“做什么”）。技能赋予它专业知识（“怎么做”），因此它们是互补的。
*   技能、MCP 和子代理和谐地协同工作。技能提供专业知识和指令，MCP 提供能力，子代理提供上下文边界（保持主代理的上下文干净）。
*   AI 开发的价值已经从更好的模型转移到了更好的基础设施。技能、子代理和多代理编排不仅仅是开发者体验的改进；它们是使代理式 AI 在规模化应用中具有经济和操作可行性的架构。

> **最终洞察：** 提示词工程仓鼠轮是可选项。是时候走出来了。
