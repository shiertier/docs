## 定义

Spec驱动开发是先把需求、边界条件、验收标准与约束写成可验证的规范，再据此推进设计、实现、评估与发布的开发方式。它强调“规范先于实现”，把 Spec 当作协作和验收的基准，而不是事后补写的文档。

## 核心内涵

Spec驱动开发的关键不是“多写一份文档”，而是把意图转成能约束后续工作的共享契约。在 AI 参与的软件工程中，这份契约既服务于人类团队对齐，也服务于 Agent 或 LLM 的执行边界，明确什么可以做、什么不能做、做到什么算完成。

当 Spec 能进一步映射到任务拆解、测试用例、Evals 或 CI 检查时，规范就不再停留在描述层，而会成为可执行的质量门槛。这也是它区别于泛泛需求说明的地方。

## 实践要点

- 先写目标、范围、输入输出、非目标与验收标准，再进入设计或编码。
- Spec 要能支撑任务拆解与回归验证，而不是只写愿景性描述。
- 与 Agent 协作时，应把约束、禁止项和完成定义显式写入 Spec，减少偏航与返工。
- 若团队已有 Evals、测试或 CI，应把 Spec 中的关键约束映射为可执行检查，避免“写了规范但无人验证”。
