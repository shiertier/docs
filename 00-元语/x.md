## 定义

用于聚合 X 平台相关文档，关注其作为公共讨论与实时分发平台的产品形态、开发者接口、社区机制与治理边界。

## 核心内涵

X（原 Twitter）既是公开信息流平台，也是面向开发者和创作者的分发基础设施。当前 X API 仍以公开帖子为核心对象，支持帖子检索、全文搜索、计数、filtered stream 和 Spaces 查询；产品侧除公开帖子外，还提供 Communities 用于围绕共同兴趣组织讨论、Spaces 用于实时音频交流。官方帮助页同时说明，Community 帖子会出现在成员主页、搜索结果和其他时间线中，公开 Space 允许任何用户加入。

它并不是一个“只负责承载内容”的中性通道。账号限制、推荐排序、限流配额与规则治理会直接影响内容的可见性、传播速度和自动化接入方式，因此在仓库语境里，X 更适合作为“平台机制 + 分发渠道 + 治理系统”的复合型元语来理解。

## 实践要点

- 进行 API 集成时，按端点读取 `x-rate-limit-limit`、`x-rate-limit-remaining` 与 `x-rate-limit-reset` 等响应头，按用户级或应用级配额设计重试、退避与缓存策略。
- 需要持续监测公开帖子时，优先考虑 filtered stream 等实时接口，而不是把高频轮询搜索当作默认方案；流式能力更适合“帖子发出后数秒内接收”的场景。
- 区分不同产品形态的用途与可见性：公开帖子适合快速分发与话题扩散，Community 帖子虽然围绕社群，但仍会进入搜索、主页和其他时间线，Spaces 适合直播式交流且默认对更广泛的听众开放。
- 不把 X 视为稳定不变的数据源；搜索、流式接口、Spaces 元数据、账号操作限制与计费/权限策略都可能随产品与政策调整而变化。
- 将平台规则纳入运营与自动化设计前提。官方规则明确禁止平台操纵、垃圾信息、冒充与违法行为，违规内容还可能被降权、限制传播或要求删除。
- 涉及公开音频或社区运营时，默认按“更高可见性、更低隐私预期”处理，因为公开 Space、Community 帖子及部分元数据都可能暴露在更广泛的公开界面中。
