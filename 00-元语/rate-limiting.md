## 定义

rate-limiting 指对请求频率、并发度或资源配额进行限制与调度的机制，用于保障系统稳定性、隔离滥用流量、维持公平性并控制成本。

## 核心内涵

- 限流不是单纯的“超过阈值就拒绝”，而是对系统承载能力做显式治理；治理对象可以是请求数、token 用量、并发任务数、数据库查询数或写入配额。
- 限流必须先定义“按谁限”和“限什么”：可按 IP、用户、租户、接口、资源类型或功能等级计量，否则很容易出现保护失衡或误伤正常流量。
- 在 API 语境中，限流既是容量控制问题，也是协议契约问题；客户端是否能根据 `429 Too Many Requests`、`Retry-After`、幂等语义和退避策略做正确响应，直接决定限流是否会演变成重试风暴。

## 实践要点

- 常见算法包括固定窗口、滑动窗口、漏桶和令牌桶，不同算法在突发流量平滑、状态存储成本与实现复杂度上取舍不同。
- 实施位置通常分为接入层、应用层和数据层，生产系统更常见的是多层限流，而不是把所有保护都压在单一点位。
- 面向外部 API 时，应把限流结果通过明确的状态码、响应头和文档约束暴露出来；但在攻击或极端过载场景下，系统也可能直接丢弃连接，而不是为每次拒绝都返回完整响应。
- 限流要与可观测性联动，至少能按租户、接口、限流原因和时间窗口观察拒绝率、排队情况与重试放大效应；在 web-crawling 等高频调度场景中，这一点尤其关键。
