## 定义

security 指通过身份、权限、数据保护、攻击防护与响应治理等手段，降低系统被滥用、被入侵、被破坏或被泄露概率与影响的整体能力。

## 核心内涵

- security 不是某个单点功能，而是覆盖需求、设计、实现、部署、运维与事件响应的持续约束。
- 它同时关心“谁能做什么”“数据如何被保护”“系统在异常和攻击下如何维持可控状态”，因此既涉及鉴权、授权、加密、密钥管理，也涉及审计、隔离、回滚与漏洞修复。
- 在本仓库语境中，security 常与 AI 系统、Agent、协议接入和自托管场景交叉出现，重点不只是防入侵，还包括防越权调用、提示词注入、密钥泄漏、供应链污染与自动化滥用。

## 实践要点

- 先明确资产、权限边界与威胁面，避免把“安全”泛化成没有对象的口号。
- 把最小权限、默认拒绝、敏感操作确认、审计留痕与密钥隔离前置到设计阶段，而不是上线后补洞。
- 把依赖治理、补丁更新、漏洞响应和发布回退纳入日常工程流程，避免把 security 误解为一次性加固动作。
- 在开放接口、Agent 工具调用与高频访问场景中，把限流、沙箱、输入校验与异常监控作为防滥用的基础控制。
- 对高风险能力明确授权前提与不可做事项，确保安全控制不会被“方便调试”或“先跑通再说”持续侵蚀。

## 边界与区分

- security 与 risk 不同：risk 负责识别威胁、损失与可逆性，security 负责把这些判断落实为防护控制与响应机制。
- security 与 compliance 不同：compliance 强调遵守法律、政策与审计要求，security 更关注技术与工程层面的真实防护能力；两者经常互相约束，但不能互相替代。
- security 与 protocol 不同：protocol 定义交互契约，security 关注这些契约是否具备鉴权、权限收口、审计与抗滥用能力。
- security 与 软件工程 不同：软件工程覆盖整体交付流程，security 是其中必须持续前置的一组质量与风险约束。
