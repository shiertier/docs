## 定义

代码审查是对代码变更进行结构化检查与讨论的协作活动，目标是在合并前确认实现是否满足功能、质量、安全与可维护性要求，而不只是找语法或格式问题。

## 核心内涵

代码审查既是质量控制，也是团队知识同步机制。它通常围绕 Pull Request 或合并请求展开，审查对象包括需求理解是否正确、设计是否合理、边界条件与回滚路径是否明确，以及测试、文档和安全约束是否齐备。在 AI coding 场景下，代码生成速度变快后，代码审查更容易成为新的工程瓶颈，因此常见做法变成“AI 初审 + 人工复核”的双层审查。

## 实践要点

- **聚焦变更意图**：先确认这次修改要解决什么问题、影响哪些模块，再进入逐行审查，避免只盯实现细节却忽略目标偏差。
- **优先高风险项**：把安全、数据一致性、兼容性、回滚难度和性能敏感路径放在最前面审。
- **区分自动化与人工审查边界**：格式、lint、基础静态检查尽量交给 CI；人工审查应集中在业务逻辑、设计权衡和异常路径。
- **建设性反馈**：评论应说明问题影响、触发条件与替代方案，减少只有结论没有上下文的“命令式反馈”。
- **控制审查规模**：鼓励小批次提交和清晰的 PR 描述，避免超大变更集稀释审查质量。
- **形成闭环**：每条审查意见都应明确为已修复、已解释或暂不处理，并保留可追溯的结论。
