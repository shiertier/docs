## 定义

web-crawling 指自动化发现、请求、解析并提取网页或站点资源的过程，核心目标是把 Web 上可访问的内容转成后续可消费的数据或文档。

## 核心内涵

Web 抓取不是单次“拿到一个页面”，而是一条持续运行的获取链路，通常包括 URL 发现、抓取调度、HTTP 请求、内容解析、去重、重试与结果落库。抓取对象既可以是 HTML 页面，也可以是 API 响应、文件、图片元数据、sitemap 与 `robots.txt` 等站点资源。

在现代网站中，抓取常常要面对 JavaScript 渲染、分页与无限滚动、登录态、反自动化策略以及大规模队列调度等问题。因此，“能在浏览器里打开页面”不等于“能稳定、低成本、可复用地完成抓取”。

## 边界与常见误区

- `web-crawling` 与 `browser-automation` 有交集，但不等同：前者强调 URL 发现、批量抓取与内容提取，后者强调驱动浏览器完成交互流程；只有在目标页面强依赖前端渲染或交互时，抓取才需要借助浏览器自动化。
- `robots.txt` 主要约束的是“能否抓取”，不是完整的访问控制或安全边界；如果要控制是否被索引，通常还需要配合页面级 `meta robots` 或 `X-Robots-Tag`。
- 被 `robots.txt` 屏蔽的 URL 仍可能因外部链接而出现在索引中，因此它不能替代鉴权、权限隔离或敏感资源保护。
- 依赖客户端 JavaScript 生成正文的页面，对支持渲染的爬虫更友好，但并不意味着所有爬虫都能正确看到内容；服务端渲染或预渲染仍然更稳妥。

## 实践要点

- 把 `robots.txt`、HTTP 状态码、站点地图与 canonical 等站点信号当作抓取入口和约束，而不是仅靠页面正文猜测抓取边界。
- 能做增量抓取时优先利用 `ETag`、`Last-Modified` 与条件请求，减少重复传输、站点压力与自己的带宽成本。
- 针对动态站点先判断是否必须执行 JavaScript，再决定是否引入无头浏览器，避免把高成本渲染误用为默认抓取路径。
- 在调度层建立限流、退避重试、去重与断点续抓，避免把临时故障放大为封禁、重复抓取或数据污染。
- 把抓取、清洗、结构化与入库解耦，确保下游 `ETL` 或 `data-pipeline` 可以独立演进和复用。
