## 定义

ui-protocol 指用于描述界面结构、组件语义、状态、事件与数据绑定规则的协议层。它不直接规定某个前端框架的实现细节，而是约束界面如何被声明、校验、传输与增量更新，使不同渲染端能够在共享语义下生成一致且可控的 UI。

## 核心内涵

UI 协议的核心在于将用户界面的视觉呈现与底层业务逻辑解耦。它通过标准化的数据结构描述组件树、属性 schema、状态字段、事件定义与数据绑定关系，使渲染引擎能够按协议动态生成、校验与更新 UI。

与通用 API 协议相比，ui-protocol 更关注“界面状态如何演进”：哪些交互可触发、触发后如何更新状态、以及渲染层如何兼容版本差异。这使它既能支撑 Web、移动端与桌面端的一致渲染，也能支撑 Server-Driven UI 等由服务端控制界面迭代的模式。

在近年的 Agent 与生成式 UI 场景中，ui-protocol 常拆成两层协同能力：一层偏向声明式 UI 结构本身，负责描述组件、绑定与渐进式更新；另一层偏向 Agent、前端应用与用户之间的双向事件流，负责表达输入、工具结果、状态同步与局部重绘边界。无论采用哪一层，核心都不是让模型直接输出任意可执行前端代码，而是让受控组件、schema 与事件流承接界面生成。

## 实践要点

- 保持协议的向后兼容性，避免字段变更或废弃导致旧版客户端渲染失败。
- 为组件、状态、事件定义可验证的 Schema，降低跨端实现歧义。
- 设计明确的事件语义与状态流转规则，避免交互行为在不同端出现不一致。
- 优化协议有效载荷体积与增量更新策略，降低传输开销并提升界面响应速度。
- 建立输入校验与降级机制，防止畸形数据导致渲染异常或崩溃。
- 明确声明式界面与可执行逻辑的边界，避免协议退化为任意代码注入或不可控渲染通道。

## 边界与区分

- ui-protocol 与 design 不同：design system 负责定义视觉语言与交互原则，ui-protocol 负责把这些约束落成可传输、可校验、可渲染的结构化描述。
- ui-protocol 与 React 等前端框架不同：React、Flutter 或原生组件库负责具体渲染，ui-protocol 负责让不同渲染端共享同一套界面语义与状态规则。
- ui-protocol 与 Agent 交互协议不同：前者聚焦组件结构、绑定和界面更新，后者更关注 Agent 状态、工具结果与用户事件如何双向流动；在生成式 UI 系统里，两者经常配合使用。
