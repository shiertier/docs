## 定义

context-database 指面向 AI 应用的上下文存储与检索层，用于在多轮交互与长任务执行中持续维护“用户、任务、环境”三类状态信息，并为模型或 Agent 按需回放其中与当前目标最相关的部分。

## 核心内涵

上下文数据库（Context Database）不是简单保存历史消息的日志库，而是把会话状态、任务进度、用户偏好、工具输出、实体关系等背景信息组织成可持续更新、可检索、可回放的外部上下文层。结合近期关于 context engineering、Agent memory 与长期记忆系统的公开实践，可以把它理解为“上下文窗口之外的长期状态底座”：模型不必在单轮提示中携带全部历史，而是根据当前任务从外部存储中选取高价值上下文，再写回新的状态变化。

它通常由多种存储形式组合而成，而不是单一数据库产品：结构化状态可放在关系型或 KV 存储中，语义召回可借助向量索引，跨实体关系可借助图结构组织。其关键不在“存了多少”，而在“写入什么、何时检索、如何淘汰、能否追溯”这四个问题是否有明确策略。

## 与相邻概念的区别

- 与 `memory` 的区别：memory 更强调系统跨会话保留与调用历史信息的能力，context-database 则是承载这类能力的存储、检索与更新基础设施。
- 与 `RAG` 的区别：RAG 主要描述“先检索、后生成”的调用链路，context-database 还覆盖状态写回、长期维护、过期淘汰与多轮任务回放。
- 与向量数据库的区别：向量检索通常只是语义召回子层；若缺少结构化状态、生命周期管理与访问策略，不能等同于完整的 context-database。
- 与 `knowledge-graph` 的区别：knowledge-graph 是组织上下文的一种结构化方式，适合表达实体关系，但不是上下文数据库的唯一实现形态。

## 实践要点

- **分层存储**：把短期工作记忆、长期档案记忆与外部知识源分开管理，避免把所有历史都直接塞进上下文窗口。
- **写入策略**：只沉淀可复用状态，如用户偏好、任务进度、关键中间结论与高价值工具结果，避免把瞬时噪音全部持久化。
- **检索策略**：围绕当前目标做召回、重排与压缩，控制进入模型窗口的上下文体积与相关性。
- **生命周期管理**：建立过期、归档、合并与删除规则，区分短期记忆与长期记忆，避免陈旧信息干扰召回。
- **一致性与隔离性**：在多用户或多 Agent 并发场景下，保证租户隔离、状态一致性与权限边界清晰。
- **可追溯性**：为写入、更新、检索与回放建立统一链路，确保“为什么召回这段上下文”可以回溯与复盘。
