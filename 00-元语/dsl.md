## 定义

DSL 是 Domain-Specific Language 的缩写，通常译为“领域特定语言”。它不是面向所有问题的通用编程语言，而是针对某一类业务、建模或配置问题设计的专用表达方式，用更受约束的语法去描述该领域里的核心对象、关系与规则。

## 核心内涵

DSL 的价值不只是“语法更短”，而是把领域约束直接写进表达层。这样一来，使用者可以用更接近业务概念或建模概念的方式工作，而不是每次都从通用语言的控制流、数据结构和底层实现细节重新搭起一层抽象。合适的 DSL 往往能提升可读性、降低沟通成本，并让渲染器、解释器或自动化流程更稳定地消费输入。

但 DSL 也有边界。它通常只适合一个相对清晰的问题域，一旦需求超出原先抽象，语言本身就可能变得难以扩展，或者重新退化成“另一门不够成熟的通用语言”。因此判断 DSL 是否值得引入，关键不在“是否新颖”，而在于它是否真正固化了高频、稳定、可复用的领域结构。

## 实践要点

- 先确认问题域是否稳定，再决定是否使用 DSL；对边界不断变化的场景，过早设计 DSL 往往会增加维护成本。
- 优先让 DSL 直接映射领域概念，而不是照搬底层实现术语，否则可读性优势很快会消失。
- 为 DSL 配套清晰的校验、错误提示和示例，避免把“语法约束”变成学习门槛。
- 在团队协作里，把 DSL 视为一种共享建模接口，而不只是某个工具内部的脚本格式。

## 边界与区分

- DSL 与通用编程语言不同：前者优先表达某个领域里的结构和约束，后者优先提供更广泛的计算能力。
- DSL 与 protocol 不同：protocol 侧重系统之间如何交换信息，DSL 侧重如何描述某一问题域里的对象和规则。
- DSL 与 markdown 不同：Markdown 的重点是文档结构与可读性，DSL 的重点是用受约束语法表达可被工具解释和执行的领域语义。
