## 定义

Git worktree 是 Git 提供的多工作目录机制，允许开发者在同一个本地仓库对象库之上，同时检出并维护多个彼此独立的工作目录。在本仓库语境里，它也常用于指代“为并行任务或多 Agent 协作划分隔离工作区”的工程实践。

## 核心内涵

- **状态隔离**：每个 worktree 拥有独立的工作区和暂存区，可以在不干扰主工作目录的情况下进行代码修改、编译或测试。
- **仓库共享**：所有 worktree 共享底层的 `.git` 目录（包括提交历史、分支、标签等对象），避免了重复克隆仓库带来的存储和时间开销。
- **并行开发**：极大地提升了多任务处理能力，开发者可以同时在多个分支上工作，无需频繁执行 `git stash` 或切换分支。
- **协作隔离**：在多 Agent 或多人并行开发场景中，worktree 常被当作独立执行单元，用于隔离分支改动、运行环境与验证过程。
- **环境解耦**：适用于需要针对不同分支维护不同依赖环境或构建产物的复杂工程场景。

## 实践要点

- **目录规划**：合理规划 worktree 的存放路径，通常建议将其放置在主仓库目录之外或特定的忽略目录中，以避免路径冲突或被误提交。
- **分支管理**：注意一个分支在同一时间只能被一个 worktree 检出，需建立清晰的分支与目录映射规范。
- **依赖同步**：在切换或创建新的 worktree 时，需注意重新安装或同步项目依赖（如 npm 包、构建缓存），因为这些文件通常不被 Git 追踪。
- **协作约定**：若把 worktree 用于多 Agent 或多任务流程，应尽量保持“一个任务对应一个 worktree”，并同步约定命名、日志查看、验证与合并前审查方式，避免只隔离目录而不隔离责任边界。
- **清理与维护**：定期使用 `git worktree prune` 等命令清理已废弃或删除的 worktree 记录，保持仓库元数据的整洁。
