## 定义

browser-automation 指通过程序化方式控制浏览器或浏览器内核，执行导航、元素交互、状态读取、网络观察与脚本注入等操作。常见控制面包括 WebDriver、WebDriver BiDi 与 Chrome DevTools Protocol（CDP）。

## 核心内涵

浏览器自动化的核心不只是“代替人点击页面”，而是把浏览器变成可远程控制的执行环境。外部程序可以建立会话，驱动页面导航、定位 DOM、监听事件、读取可访问性树或网络请求，并在需要时注入脚本。这使它成为端到端测试、动态网页抓取、业务流程自动化和浏览器型 Agent 的基础能力。

与 `web-crawling` 相比，browser-automation 更强调交互与状态变化的可控性，而不只是获取页面内容；与纯 API 调用相比，它能够处理登录、表单、多步骤操作和前端渲染后的真实界面状态，但运行成本更高、脆弱性也更强。

## 关键接口与工程特征

- 标准化控制接口通常来自 WebDriver；在需要事件流与更强实时性的场景下，WebDriver BiDi 进一步补足了双向通信能力。
- 基于 Chromium 的工具常直接使用 CDP 控制 DOM、Network、Runtime、Input、Page 等域，因此调试、录制、性能观测与自动化往往共用同一套底层接口。
- 现代自动化框架通常会把“等待页面达到可操作状态”做成默认能力，例如先确认元素可见、稳定且可接收事件，再执行点击或输入，以减少脚本脆弱性。
- 在 AI Agent 场景中，browser-automation 往往不会直接暴露原始 DOM，而是先转成可访问性快照、结构化页面状态或高层工具命令，降低模型误操作概率。

## 实践要点

- 元素定位优先使用稳定且具语义的定位方式，例如 role、label、test id 或明确的数据属性，避免把易变的样式路径当成主定位手段。
- 把等待条件视为脚本的一部分，而不是临时补丁；对导航完成、元素可操作、网络空闲和异步渲染要分别建模。
- 区分测试、抓取和 Agent 操作三类目标：测试关注可复现与断言，抓取关注吞吐与完整性，Agent 操作关注权限边界、人工接管和失败恢复。
- 浏览器会话常含登录态、Cookie、Header 与本地存储，自动化产物不应直接入库或泄露给无权限方。
- 面向外部站点运行自动化时，仍需遵守访问规则、频率限制与授权边界，避免把“可自动化”误当成“可无限制执行”。
