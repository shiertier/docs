## 定义

`skills` 指一组可被复用、可迁移的能力封装。在本仓库中，这个词有两层常见语义：一是人的专业能力；二是 AI Agent 语境中的 Agent Skills，即以 `SKILL.md` 为入口、连同脚本、模板和参考资料一起按需加载的能力包。后者是本仓库更常见的用法。

## 核心内涵

- 对人而言，skills 是可持续积累、可迁移到不同任务或职业阶段的能力集合。
- 对 Agent 而言，skills 不是一次性提示词，而是把领域知识、步骤约束、工具使用方法与可执行资源组织成可复用单元。
- Skills 的工程价值在于渐进式加载：Agent 先看到名称和描述，真正命中后再读取详细指令与附加资源，减少上下文噪音。
- Skills 适合沉淀“反复出现、需要稳定做法、又不值得单独做成完整产品”的任务场景。

## 实践要点

- 先从重复出现的真实任务中提炼 skill，不要为假设需求提前封装“巨无霸”技能。
- 将 `name`、`description` 写成清晰的触发信号，把 `SKILL.md` 主体聚焦在步骤、边界和完成标准上。
- 让单个 skill 保持可组合：知识说明、脚本、模板与参考资料按需拆分，便于和其他 skills 或 workflow 组合。
- 用真实失败案例持续迭代，并通过评估或回归检查确认 skill 的触发、执行步骤和输出质量没有退化。

## 边界与区分

- `skills` 与 `prompt` 不同：prompt 更像一次性输入；skills 是可发现、可复用、可迭代的封装。
- `skills` 与 `tool` 不同：tool 负责具体执行；skills 负责告诉 Agent 何时用什么工具、按什么约束完成任务。
- `skills` 与 `workflow` 不同：workflow 强调节点编排与状态流转；skills 更强调某类任务的知识、步骤和最佳实践，可作为 workflow 的节点或局部能力。
- `skills` 与 `mcp` 不同：mcp 解决对外系统连接与协议标准化；skills 更适合封装本地流程、领域方法和按需加载的上下文资源。

## 风险与注意事项

- Skills 往往会引导 Agent 读写文件、运行脚本或调用外部工具，只应安装和执行来自可信来源的 skills。
- 把仓库、长文档或大量规则无差别塞进单个 skill，会抬高触发噪音并削弱模型判断力。
- 如果某项能力只是一次性提示或临时流程，不一定要升级为 skill；过度封装会增加维护成本。
- 在人类职业语境下讨论 skills 时，要避免把“工具熟练度”误当成全部能力结构，仍需结合学习路径、决策与实践反馈来评估。
